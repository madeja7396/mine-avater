# 開発ガイドライン

## 目的

実実装前に、仕様主導で安全に並列開発できる土台を維持する。

## 守るべき順序

1. `specs/` 更新
2. テスト更新
3. 実装更新
4. `eval` 更新

## 設計原則

- 仕様は `specs/` をSSOTにする。
- 変更はタスクロックで排他管理する。
- 実装単位は独立可能な小粒度に保つ。
- ログは機械可読フォーマットを維持する。

## 変更管理

- 中間成果物を変更する場合は、契約更新 + テスト更新を同時実施する。
- `fast` が壊れる変更は優先的に修正し、`full` で最終確認する。
- 閾値の変更は理由をPR本文に明記する。
